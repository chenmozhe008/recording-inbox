# recording-inbox 产品定位、目标用户与内容总纲

> 用途：这份文档是 GitHub 首页、安装教程、案例文章、公众号选题和后续产品取舍的共同依据。
>
> 更新时间：2026-07-12。本文描述的是当前开源仓库的实际能力，不把规划中的功能写成已经可用的功能。

## 一句话定位

**recording-inbox 是面向高频录音用户的“录音转写、文字稿和智能纪要工作台”。**

用户用手机原生录音、录音笔、会议设备或任何能导出音频的工具完成采集；把音频上传到飞书 `inbox` 后，一台已有的 Mac 或 Windows 电脑自动完成本地转写、智能纪要、飞书归档和完成通知。

它不是另一款录音 App，也不要求用户购买新的录音硬件。知识库沉淀是处理完成后可以继续获得的价值，不应掩盖第一需求：**把大量录音稳定地变成可读、可找、可交付的文字稿和纪要。**

## 用户首先遇到的问题

录音本身并不难，困难从录音结束后才开始：

- 录音很多，单条时长长，飞书妙记或其他云服务的免费分钟数不够用。
- 文件散在手机、微信、网盘、录音笔或电脑里，之后难找、难命名、难批量处理。
- 两小时会议的完整文字稿需要保留，但人真正要看的又是结论、章节、待办和关键引用。
- 不同行业需要不同整理方式：课程、访谈、项目会、客户沟通不应使用一套固定纪要。
- 结果常常留在另一个转写平台，团队真正协作却发生在飞书里。
- 失败、电脑关机、网络中断或重复上传后，用户不希望重新从头处理。

recording-inbox 的主张不是“录得更好”，而是：

> **录完以后，不必反复听、不必手工整理、不必把成果搬来搬去。**

## 标准工作流

```text
手机原生录音 / 录音笔 / AI 录音硬件 / 电脑音频
                 ↓
          上传到飞书 inbox
                 ↓
      Mac 或 Windows 本地处理电脑
        本地语音转写 + AI 智能纪要
                 ↓
    飞书双文档：智能纪要 + 完整文字稿
                 ↓
       飞书消息通知，打开结果查看
```

### 必要条件：需要一台电脑

需要一台可运行处理程序的 **Mac 或 Windows 电脑**。它可以是工作笔记本、办公室电脑或家用电脑，不需要专门购买服务器。

- 手机负责录音和上传，不需要长期连接电脑。
- 电脑开机并联网时，自动处理飞书 `inbox` 中的新文件。
- 电脑临时关机时，音频仍保留在飞书 `inbox`；下次开机后再处理。
- 这是一项明确的使用条件，不应在宣传中隐藏。它换来的是本地转写、长期批量处理和可自定义的模型与文稿结构。

## 核心价值

### 1. 高量录音的长期处理能力

适合持续产生会议、课程、访谈和沟通录音的人。用户不必把全部处理能力绑定在某一家转写产品的分钟额度、订阅套餐或硬件生态上。

### 2. 一条录音同时交付“智能纪要”和“文字稿”

最终交付不是只有一段摘要，而是两篇独立飞书文档：

1. AI 生成标题
2. 智能纪要
3. 主题大纲
4. 待办
5. 智能章节
6. 关键决策和引用
7. 完整文字稿（单独文档）

用户既能两分钟了解会议，也能在需要时回到原始表述核对。

### 3. 不绑定录音入口和设备

手机、传统录音笔、AI 录音设备、微信下载的音频、电脑本地文件都可以作为音频来源。产品的价值在后处理，而不是要求用户更换已经习惯的录音方式。

### 4. 文稿逻辑由用户控制

默认模板沿用经过真实使用调整的通用智能纪要，首次安装不需要做选择；另有客户、访谈、播客、课程、培训、项目、调研、复盘和口述模板。模型、提示词和 API 服务也可替换，让固定场景逐步形成自己的输出结构、术语和待办规则。

默认推荐 DeepSeek V4 Flash：当前按 token 计费很低，常见纪要调用的模型费用通常可以忽略。项目同时支持其他 OpenAI 兼容服务，不把用户锁在单一模型或 API Key 上。

### 5. 结果回到飞书

飞书既是收件入口，也是最终查看和协作的位置。用户不用在录音 App、转写 SaaS、聊天软件和文档工具之间搬运结果。

## 目标用户

以下不是抽象的“所有需要录音的人”，而是最值得优先服务和传播的具体人群。

| 优先级 | 用户群 | 高频场景 | 核心痛点 | 应强调的价值 |
| --- | --- | --- | --- | --- |
| 最高 | 项目经理、咨询、销售、客户成功、服务型团队 | 客户会、项目会、方案沟通、复盘、跨部门同步 | 会多、结论散、待办易漏、飞书是协作中心 | 自动产出文字稿、结论和待办，直接回到飞书 |
| 最高 | 访谈和内容从业者 | 用户访谈、人物采访、播客、选题沟通、素材整理 | 录音需要完整留存，之后要检索、引用、改稿 | 保留完整文字稿，同时给出结构化提要 |
| 高 | 培训、课程、教研人员 | 内训、讲座、课程、教研讨论、复盘分享 | 单条长、信息密、学员或团队要反复查阅 | 章节化文字稿、提纲、课程模板与长期归档 |
| 高 | 高频使用飞书的中小团队 | 例会、业务沟通、远程录音、复盘 | 不希望成果留在另一套 SaaS 中 | 以飞书为收件、查看和通知中心 |
| 中 | 个体创业者、自由职业者 | 客户沟通、需求确认、灵感口述、项目讨论 | 不想长期购买大量转写时长，希望按自己方式整理 | 可控成本、可换模型、可改模板 |
| 中 | 调研、田野和行业研究人员 | 座谈、走访、深度访谈、内部讨论 | 原话需留存，后续需要反复搜索和引用 | 可追溯的完整文字稿与结构化纪要 |

### 不应优先吸引的人

- 每月只有少量短录音，只想立即点一下生成结果的人。
- 不用飞书，或不愿意在电脑上做一次初始配置的人。
- 需要会中实时字幕、实时总结、实时多语翻译的人。
- 无法确认录音授权、数据合规和本地/云端数据边界的高敏感行业用户。

## 与替代方案的关系

| 方案 | 最强之处 | recording-inbox 的位置 |
| --- | --- | --- |
| 手机原生录音 | 稳定、离线优先、用户已有、无需学习 | 推荐继续使用；本项目解决录完以后的转写与整理 |
| AI 录音硬件 | 一键采集、便携、常附带自动同步与总结 | 不与硬件争采集；硬件导出的音频也可进入本流程 |
| 飞书妙记 | 飞书原生体验、回放、实时能力、协作顺滑 | 适合轻量和即时需求；本项目适合高频、长录音和自定义处理逻辑 |
| IMA 等录音/知识应用 | App 内录音到总结的路径短，有独立知识库 | 适合个人即时记录；本项目更强调飞书归档、批量处理和开放模型选择 |
| 纯转写 SaaS | 转写、编辑与消费级界面成熟 | 本项目的区别是自动完成“转写后整理并回到飞书”，而非只给一个转写页面 |

对 AI 硬件的正确表述是：

> 不需要替换你的录音设备。只要能导出音频，recording-inbox 就能接管后续的转写、文字稿和智能纪要。

不要把产品写成“替代录音笔”或“比所有录音硬件更好”。硬件擅长采集，软件工作流擅长处理和归档，两者可以互补。

## 当前已实现的能力

- 飞书云盘 `inbox` 收取常见音频/视频文件。
- 本地 FunASR/SenseVoice 转写，支持可选 `whisper.cpp` 兜底。
- 输出 `transcript.txt`、结构化转写数据和飞书文档。
- 用兼容 OpenAI API 的模型生成智能纪要；可配置模型、模板和自定义提示词。
- 两篇最终飞书文档：一篇智能纪要，一篇完整文字稿；智能纪要反链文字稿。
- 单人录音尽量避免无意义的“说话人 1”；多人可保留编号区分。
- 处理状态、去重、失败重试、电脑重启后继续和已处理文件归档。
- 飞书完成/失败通知；通知消息以简短状态和结果链接为主。

## 当前尚未实现或不应宣传为已实现

- 不会自动从 iPhone 系统语音备忘录直接取文件；当前手机上传仍需要用户把音频传到飞书 `inbox`。
- 不提供独立的原生手机录音 App，也不直接对接蓝牙、USB、SD 卡或某一品牌 AI 录音设备。
- 通知默认只发送一条完成消息，不分别推送两次；消息提供“打开智能纪要”和“打开文字稿”两个入口。
- 没有按每条录音在手机端选择模型、模板或场景的界面；配置以全局设置为主。
- 没有可靠地把“说话人 1/2”自动映射为真实姓名。
- 没有自动把结果创建为飞书知识库/Wiki 节点；当前是输出到指定飞书云盘文件夹。
- 没有把多段同主题录音自动合并成一份纪要。
- 不提供会中实时转写或实时摘要。
- 智能纪要开启后，文字记录会发送给用户配置的 LLM API；不能宣传为“所有内容永不出电脑”。

## 以后优化功能时的优先级

### P0：降低“录完以后送进来”的摩擦

1. iPhone 原生录音的快捷上传路径：收藏 inbox、分享菜单、快捷指令或经过验证的文件流程。
2. 通用本地收件箱：让电脑上的 iCloud 文件夹、下载目录或录音设备导出目录也能自动进入队列。
3. 更清楚的完成通知：一个结果页中明确跳转到“智能纪要”和“完整文字稿”。

### P1：让内容更符合业务

1. 按文件夹、标签或场景自动选会议/课程/访谈/项目模板。
2. 可维护的专有名词表与纠错规则。
3. 原音频入口、标题、日期和来源在最终文档中完整可追溯。
4. 可选的飞书知识库归档。

### P2：只在真实需求出现后再做

1. 特定硬件品牌的直连同步。
2. 多段音频的自动主题聚合和合并。
3. 原生手机录音 App、实时字幕和会中总结。

这些能力成本高，且会使项目从“开放的后处理工作流”偏向“另一个录音平台”；不应抢在上传和文稿体验之前做。

## 对外文案口径

### 推荐的一句话

> 你照常用手机或录音设备录音，把文件放进飞书 inbox；一台已有电脑会把它变成完整文字稿和智能纪要，再发回飞书。

### GitHub 首页标题方向

> 大量录音，自动变成飞书里的文字稿和智能纪要

### 适合文章开头的表述

> 录音的问题从来不是“能不能录下来”，而是录完以后：两小时的音频没人回听，妙记额度不够，待办散在聊天里，文字稿又留在另一个平台。recording-inbox 用一台已有电脑，把任何来源的音频持续处理成飞书里的文字稿和智能纪要。

### 必须保留的诚实说明

> 处理电脑需要在需要转写时开机联网。它可以是普通工作笔记本；电脑关机期间，音频先留在飞书 inbox，开机后继续处理。

## 内容与案例选题

优先使用真实、具体的工作日场景，而不是泛泛讲“AI 提效”。

1. **客户会很多的项目经理**：一天 5 场会，如何把录音变成可确认的结论和待办。
2. **自媒体访谈**：一小时访谈如何同时获得可引用文字稿和选题提纲。
3. **课程与内训**：一节两小时课如何整理成章节、知识点与复习材料。
4. **飞书团队的低成本替代路径**：妙记额度不够后，如何继续保留飞书里的工作体验。
5. **已有录音设备的人**：不换设备，如何把设备输出接入自己的转写和纪要流程。

每个案例都应回答四件事：录音来自哪里、为什么原方案不够用、最终得到什么文档、用户之后如何在飞书中使用它。
